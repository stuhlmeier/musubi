Detailed Description

An application window.

Windows should handle their own lifecycle; they are destroyed when the associated object is deleted.

A screen can be attached to handle updates and render content.

Windows created through an application instance will be automatically registered, as will their associated input_pollers. Once registered, input is polled once per tick and passed to all windows via update().

For a window implementation to be registrable by an application, it may define a member typedef poller_type.

◆ id_type

using musubi::window::id_type = uint32

The unique ID type used internally to identify windows.

Window identifiers must be globally unique, regardless of what framework managed or created them. Since many libraries return an internal ID that does not necessarily respond to the actual system window handle, simply using the respective framework window ID may result in identifier collisions.

Sets this window's current screen.

The old screen instance is destroyed.

Window implementations should extend the lifetime of the current screen to after the current tick has finished, so that setting a new screen from the current screen's update callback will not destroy the screen before it is finished.
